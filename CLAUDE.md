This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 仓库性质

个人技术笔记仓库，**纯 Markdown，无构建 / 测试 / 依赖**。不要尝试 `npm install`、`build`、`test` 等命令——没有 `package.json`，也不需要。

工作范围限定在：编辑现有 `.md`、新建 `.md`、调整目录结构、用 `git` 提交。

## 笔记写作约定

仓库内已有约定俗成的两类笔记格式，新增内容时请保持一致。

### 1. 工具 / 库清单类（如「前端生态库推荐」「Mac 工具软件」）

- **基本格式**：`- [名称](链接)：一句话简介。`
- **⭐ 标记**：放在**链接右括号外**，前面带一个空格，表示作者首选或重点推荐：
  - ✅ `- [unhead](https://...) ⭐：描述……`
  - ❌ `- [unhead ⭐](https://...)`（旧写法，链接内）
  - ❌ `- [unhead](https://...)⭐：……`（旧写法，外部无空格）
  - 后续遇到旧写法**可顺手统一为外部带空格形式**。
- **二级要点**：用缩进列表补充「特点 / 适用场景 / 注意」，例如：
  ```
  - [decimal.js](...) ⭐：高精度数字计算库……
    - 注意：金额计算强烈推荐统一封装金额工具层。
  ```
- **避免使用项目 README 的营销话术**作为唯一描述（如「快 N 倍 / 体积小 N%」），尽量补一句作者视角的判断或场景。

### 2. 代码封装类（如「Axios 封装」「LocalStorage 封装」）

- 顶部用「**亮点**：」列出 3–5 条特性；
- 主体是带语言标记的代码块（` ```JavaScript `、` ```TypeScript `、` ```Vue `）；
- 不要把整段代码替换成解释——这类笔记的价值就是可直接复制使用的成品。

### 通用规则

- 文件名 / 目录名使用**中文 + 空格**（如 `前端基础能力/`），不要改成短横线或拼音。
- 章节使用 `##` 二级 / `###` 三级标题分类；同一文件内层级一般不超过 4 级。
- 链接首选 GitHub 仓库或官网；避免使用 GitHub 自动生成的长 title 作为链接文字（形如 `GitHub - org/repo: 项目描述……`），改写为简洁的项目名。

## 提交信息约定

参考已有 commit 风格，使用 conventional commits 中文描述：

- `docs: 重构并整理前端生态库推荐文档`
- `chore: 规范前端基础能力文档命名`
- `docs(markdown): 整理重构Mac开发文档并补充新内容`

`docs:` 用于内容增删改；`chore:` 用于重命名 / 目录调整 / 删除等不影响内容的整理。

## 编辑时的常见任务

- **新增一个工具推荐**：定位到对应分类的 `###` 小节，按上方清单格式追加一行；判断是否值得加 ⭐。
- **新增一个分类**：在对应文件中加 `###`（极少数情况下加 `##`），保持与同级章节风格统一。
- **新增一篇笔记**：放在最贴近的现有目录下；若需要新分类目录，先与用户确认命名。
